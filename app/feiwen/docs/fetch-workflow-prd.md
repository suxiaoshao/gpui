# feiwen 抓取页优化 PRD

关联 issue: https://github.com/suxiaoshao/gpui/issues/105

参考图: [assets/fetch-workflow-reference.png](assets/fetch-workflow-reference.png)

## 背景

当前 feiwen 抓取页只提供基础输入框和单行状态文案。长时间抓取时，用户很难判断当前任务运行到哪一页、已经完成多少页、入库了多少条数据，以及失败发生在什么位置。失败时页面只显示粗略错误类型，无法明确失败页码和错误详情；抓取任务也不支持用户主动中断、从中断状态继续、或从失败页重新开始。

本阶段只产出产品与视觉参考，不实现代码、不调整数据库结构、不修改抓取逻辑。

## 目标

- 优化抓取表单，使入口链接、页码范围和 Cookie 状态更清晰。
- 优化抓取信息展示，让不同状态下的主信息、详情和可用操作有明显差异。
- 失败时必须展示失败页码、错误类型、错误详情和已完成进度。
- 支持后续实现抓取中断、从中断处继续、从失败页重试。
- 明确后续抓取逻辑要求：失败应立即展示并停止当前任务，不应静默失败后继续抓取下一页。

## 用户体验

### 抓取配置

抓取表单应作为独立区域展示，包含：

- 入口链接
- 起始页
- 结束页
- Cookie
- Cookie 可见性状态，例如“Cookie 已隐藏”
- 主操作按钮“开始抓取”

Cookie 不应在主界面中裸露完整长文本。默认展示为隐藏状态，用户只需要知道当前是否已填写。

### 任务状态

任务状态区域应展示当前任务的核心进度：

- 当前状态：未开始、正在抓取、中断、抓取失败、抓取成功
- 当前页和总页数，例如“当前页 18 / 5201”
- 已完成页数
- 入库总数
- 开始时间或最近更新时间
- 当前状态说明

运行中状态应突出“当前正在处理哪一页”。成功状态应突出完成页数和最终入库总数。中断状态应突出已保留进度和下一次继续页。失败状态应突出失败页、错误类型和错误详情。

### 操作与恢复

不同状态下展示不同操作：

- 未开始：开始抓取
- 正在抓取：中断
- 中断：从中断处继续
- 失败：从失败页重试
- 成功：可重新开始或回到查询

恢复操作的含义：

- 从中断处继续：从最后成功页的下一页继续。
- 从失败页重试：从失败发生的页码重新开始，不跳过失败页。

### 页面日志

页面底部应展示逐页日志，帮助用户理解长任务过程：

- 页码
- 状态
- 入库数量
- 耗时
- 信息摘要
- 详情操作

日志至少覆盖成功、抓取中、失败三类状态。失败行应使用明显的错误样式，并展示失败原因摘要。

## 状态设计

### 未开始

显示抓取配置和空状态说明。用户可以填写表单并开始抓取。

### 正在抓取

显示当前页、总页数、已完成页数、入库总数和进度条。操作区只提供“中断”等与当前任务相关的动作，避免同时展示与当前状态无关的恢复动作。

### 中断

显示已完成页数、最后成功页、下一次继续页和保留的任务配置。主操作为“从中断处继续”。

### 抓取失败

显示失败页、错误类型、错误详情、已完成页数和入库总数。页面必须说明任务不会继续静默抓取下一页。主操作为“从失败页重试”。

### 抓取成功

显示完成范围、完成页数、最终入库总数和完成时间。用户可以回到查询页或重新配置下一次抓取。

## 非目标

- 本阶段不修改 Rust 代码。
- 本阶段不调整数据库 schema。
- 本阶段不实现跨应用重启后的任务恢复。
- 本阶段不改变现有解析规则或请求 headers。
- 本阶段不新增依赖。
